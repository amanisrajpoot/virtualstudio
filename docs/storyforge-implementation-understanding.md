# StoryForge Implementation Understanding

Source reviewed: `C:\Users\aman\Downloads\StoryForge Implementation Blueprint v1.pdf`

This document is a working reference for later implementation. It does not start
implementation, scaffold the app, or create runtime code.

## Core Mission

StoryForge is a deterministic story-to-video platform powered by Godot.

The product should feel like AI video generation to the user, but the actual
cinematic output must come from reusable assets, semantic world data,
intent-driven actors, and a rules-based Director Engine.

The AI layer is intentionally narrow:

- Understand a natural-language story.
- Convert it into structured story instructions.
- Optionally generate dialogue.

The AI layer must not decide:

- Camera placement.
- Character placement.
- Animation selection.
- Lighting.
- Physics.
- Rendering.

Those choices belong to deterministic systems: validators, semantic locations,
intent mappings, director rules, templates, timeline execution, and the Godot
runtime.

## Target User Flow

The user writes a short story prompt, for example:

`Halku is selling bottled moonlight in a village market. A policeman becomes suspicious and questions him.`

The system should:

1. Understand the story.
2. Choose a supported scene.
3. Choose supported characters.
4. Place characters using semantic anchors.
5. Choose required props.
6. Generate story beats and dialogue.
7. Convert beats into a deterministic runtime timeline.
8. Direct cameras through rule-based presets.
9. Execute actions in Godot.
10. Render a vertical `1080x1920` MP4 for Shorts, Reels, and TikTok.

## Architecture Chain

Intended pipeline:

```text
User Story
-> Story Compiler
-> Story DSL
-> Validator
-> Director Engine
-> Runtime DSL
-> Godot Runtime
-> Video Renderer
-> MP4 Output
```

The important architectural boundary is between `Story DSL` and `Runtime DSL`.

- `Story DSL` describes story intent in human-readable structured form.
- `Runtime DSL` describes executable instructions for Godot.
- The Director Engine is the only system that should translate intent into
  staging, action mappings, camera goals, and timing.

## Proposed Technology Stack

- Frontend: Next.js, React, Tailwind.
- Backend: FastAPI, Python 3.12.
- Database: PostgreSQL.
- Queue: Redis.
- Storage: S3-compatible object storage.
- Rendering: Godot 4.
- AI providers: provider-agnostic, with possible support for OpenAI, Gemini,
  Claude, and Ollama.
- Deployment: Docker first, Kubernetes later.

## Phase 1 Scope

Phase 1 is the core runtime.

Goal:

- Execute structured stories.
- Avoid AI complexity.
- Avoid procedural generation.
- Avoid custom characters.

Success condition:

- A valid Story DSL document can render a video through the deterministic
  runtime pipeline.

Phase 1 asset library:

- Scenes: `village_market`, `village_house`, `police_station`, `classroom`.
- Characters: `halku`, `policeman`, `shopkeeper`, `teacher`, `student`,
  `villager_male`, `villager_female`.
- Props: `tree`, `buffalo`, `stall`, `chair`, `table`, `bottle`,
  `blackboard`, `desk`.
- Animations: `idle`, `walk`, `run`, `talk`, `laugh`, `cry`, `point`,
  `celebrate`, `inspect`, `sit`, `stand`, `pickup`, `throw`.

## Story DSL Responsibility

The Story DSL is the compiler output and should stay close to story intent.

It may contain:

- Scene identifier.
- Character identifiers.
- Beat list.
- Beat intent.
- Actor.
- Target.
- Props or semantic references.
- Dialogue text, if generated or supplied.

It must not contain:

- Coordinates.
- Camera angles.
- Raw animation instructions.
- Godot-specific node paths.

Example shape:

```json
{
  "scene": "village_market",
  "characters": ["halku", "policeman"],
  "beats": [
    { "intent": "sell_product", "actor": "halku" },
    { "intent": "inspect", "actor": "policeman", "target": "halku" }
  ]
}
```

## Runtime DSL Responsibility

The Runtime DSL is generated by the Director Engine and should be executable by
the Godot runtime.

It may contain:

- Scene identifier.
- Timeline events.
- Actor actions.
- Semantic movement targets.
- Camera events.
- Audio events.
- Timing and delays.
- State updates.

Example shape:

```json
{
  "scene": "village_market",
  "timeline": [
    {
      "time": 0,
      "actor": "halku",
      "action": "walk_to",
      "target": "stall_front"
    }
  ],
  "camera": [
    { "time": 0, "preset": "wide_market" }
  ]
}
```

## Main Module Boundaries

### Compiler

Input: natural-language story.

Output: Story DSL.

Responsibilities:

- Extract scene.
- Extract characters.
- Extract intents.
- Extract or generate dialogue, if enabled.

Hard boundary:

- The compiler must not emit coordinates, camera angles, or animation choices.

### Validator

Input: Story DSL.

Responsibilities:

- Confirm the scene exists.
- Confirm characters exist.
- Confirm intents exist.
- Confirm props exist.
- Reject invalid stories.
- Return actionable correction suggestions.

### Asset Registry

Responsibilities:

- Store known scenes, characters, props, animations, voices, and later packs.
- Provide stable IDs used by DSLs and validators.
- Keep runtime assets discoverable without exposing raw Godot scene editing to
  users.

### Semantic World System

Every location has named anchors instead of manually authored coordinates.

Example anchors for `village_market`:

- `entrance`
- `market_center`
- `stall_front`
- `stall_left`
- `buffalo_area`
- `tree_area`

Actors move between anchors. The Director Engine chooses anchors and
relationships; authors and AI do not write coordinates.

### Actor System

Base actor capabilities:

- `move_to()`
- `speak()`
- `look_at()`
- `perform_intent()`
- `play_action()`
- `update_state()`

Actor state:

- Emotion.
- Attention.
- Energy.
- Current location.
- Inventory.

### Intent System

Runtime behavior should be driven by intents, not direct animation calls.

Initial intents include:

- `sell_product`
- `question`
- `inspect`
- `argue`
- `laugh`
- `run_away`
- `chase`
- `celebrate`
- `greet`

The Director maps an intent to staging, actor actions, camera goals, and
emotional state.

### Director Engine

This is the central module.

Input: valid Story DSL.

Output: Runtime DSL.

Responsibilities:

- Staging.
- Distances.
- Facing directions.
- Camera goals.
- Lighting presets.
- Intent-to-action mappings.
- Timeline construction.

Example rule interpretation:

```text
Intent: argue
Director result:
- distance = confrontation
- face_each_other = true
- emotion = angry
- camera = shot_reverse_shot
```

### Camera Director

Camera selection is rules-based.

Camera goals include:

- `show_reaction`
- `show_dialogue`
- `show_argument`
- `show_chase`
- `show_introduction`

Camera presets include:

- `wide`
- `closeup`
- `medium`
- `tracking`
- `reaction`
- `hero`
- `dialogue`
- `shot_reverse_shot`

### Timeline System

The timeline executes Runtime DSL events.

It should support:

- Sequential actions.
- Parallel actions.
- Delays.
- Camera events.
- Audio events.
- Future nesting.

### World State System

Persistent memory for episodic storytelling.

Example:

```json
{
  "halku": {
    "inventory": ["moonlight_bottle"]
  }
}
```

World state should update after each rendered video.

### Dialogue System

Dialogue can be AI-assisted but should still be driven by intent and actor
context.

Example:

```text
Intent: question
Actor: policeman
Dialogue: What exactly are you selling?
```

### Voice System

Phase 1 uses predefined voices.

Example mappings:

- `halku` -> `cheerful_male`
- `policeman` -> `stern_male`
- `teacher` -> `strict_female`

Voice cloning is future scope.

### Rendering Pipeline

Expected runtime sequence:

1. Load Runtime DSL.
2. Spawn scene.
3. Spawn actors.
4. Place props.
5. Execute timeline.
6. Capture camera.
7. Render frames.
8. Encode MP4.
9. Return URL.

### Godot Plugin

Godot plugin panels should include:

- Stories.
- Characters.
- Locations.
- Props.
- Timeline.
- Render.
- World State.
- Compiler.
- Director.

Users should not interact with raw Godot scenes directly.

## Phase 1 Definition Of Done

The platform should accept a story equivalent to:

`Halku sells bottled moonlight. A policeman questions him.`

Within roughly 60 seconds, it should:

- Understand or accept structured story intent.
- Select the supported scene.
- Stage supported characters.
- Generate or attach dialogue.
- Direct cameras through rules.
- Render a vertical MP4.

The user must not need to:

- Edit a scene manually.
- Edit a timeline manually.
- Pick animations manually.
- Place cameras manually.

## Guardrails For Future Implementation

- Start with deterministic structured input before AI.
- Treat AI output as untrusted and validate it.
- Version both Story DSL and Runtime DSL from the beginning.
- Keep Godot runtime execution separate from story understanding.
- Prefer semantic anchors over coordinates in external contracts.
- Avoid procedural generation until the base asset registry and runtime are
  stable.
- Avoid custom characters until the character generator phase.
- Build renderer and queue boundaries so server-side rendering remains the
  default path.
- Keep provider-specific AI code behind a provider-agnostic interface.

- Which S3-compatible storage should be used locally: MinIO or filesystem-only
  development storage?
- What job status model should the render queue expose?
- What database tables are needed for stories, render jobs, assets, world
  state, and outputs?
- How should Godot assets be represented in the registry: paths, resource IDs,
  packed scenes, or all of these?
- What is the minimum viable asset set available locally for Phase 1?
- Should dialogue be generated in Phase 1 or provided by static templates until
  the runtime is stable?
- Which TTS engine should be used for predefined voices?
- What is the exact acceptable render time for a first alpha?
